## Data Architecture

Data flows from operational systems into dashboards through a coordinated data architecture. Clean, reliable data enables good dashboards. Poor data architecture undermines even well-designed dashboards.

**Core Principle**: Single source of truth for all metrics. Consistency across dashboards. Regular data validation.

## Data Sources

### Operational Systems

**Systems that feed dashboards**:
- Finance system (revenue, costs, profitability)
- HR system (headcount, engagement, retention)
- Project management system (schedules, completion rates)
- Customer management system (satisfaction, retention)
- Operations system (quality, efficiency, cycles)
- Communication systems (team engagement signals)

**Characteristics**:
- Current transactional data
- Daily or real-time updates
- Accurate source of record
- System-specific metrics

### Data Warehouse or Lake

**Central data repository**:
- Aggregates from multiple sources
- Cleaned and normalized data
- Historical data for trends
- Pre-calculated metrics
- Single source of truth

**Design**:
- Dimensional data model
- Fact tables (metrics)
- Dimension tables (context)
- Incremental updates
- Data quality checks

## Data Integration

### ETL Processes

**Extract, Transform, Load**:
- Extract data from source systems
- Transform into standard format
- Validate for accuracy
- Load into data warehouse
- Track process status

**Frequency**:
- Real-time or near real-time (critical metrics)
- Hourly or 4-hourly (operational metrics)
- Daily (standard metrics)
- Weekly or monthly (analytical metrics)

**Reliability**:
- Error handling and retry logic
- Data validation and quality checks
- Alerts on process failure
- Recovery procedures
- Audit trail

### Data Quality

**Quality dimensions**:
- Accuracy (data is correct)
- Completeness (all data captured)
- Consistency (same data consistent across systems)
- Timeliness (current, not stale)
- Uniqueness (no duplicates)

**Quality checks**:
- Row counts (expected volume?)
- Value ranges (data within bounds?)
- Relationships (referential integrity?)
- Calculations (math correct?)
- Nulls (expected missing data?)

**Quality issues**:
- Detection and alerting
- Root cause analysis
- Remediation procedures
- Data governance
- Prevention focus

### Data Governance

**Who owns what**:
- Data steward (each data domain)
- Quality responsibility
- Definition authority
- Change control
- Access decisions

**Policies**:
- Data naming standards
- Calculation methodology
- Update frequency
- Retention periods
- Archive procedures

## Metric Definitions

### Metric Specification

**For each metric define**:
- Name (clear, consistent)
- Definition (what does it measure?)
- Calculation (how is it computed?)
- Data source (where does data come from?)
- Update frequency (when is it updated?)
- Responsible party (who owns it?)
- Units (what are we measuring?)
- Context (what does it mean?)

### Calculation Methods

**Types of calculations**:
- Simple aggregation (sum, average, count)
- Ratios (percentage, efficiency)
- Derived metrics (multi-step calculations)
- Composite metrics (combining multiple measures)
- Trend calculations (comparing periods)

**Standardization**:
- Consistent formulas across organization
- Document all calculations
- Version control for changes
- Business rule clarity
- No ad hoc metrics

### Metric Catalog

**Maintain catalog of**:
- All metrics used in dashboards
- Definition for each metric
- Calculation method
- Responsible owner
- Last updated date
- Related metrics
- Drill-down details

**Purpose**:
- Single reference point
- Prevent duplication
- Ensure consistency
- Enable onboarding
- Support maintenance

## Real-Time vs. Batch

### Real-Time Metrics

**For critical operational metrics**:
- Event-driven updates
- Streaming data pipeline
- Near-immediate availability
- Limited historical
- High complexity

**Examples**:
- System uptime
- Critical issues
- Order status
- Queue length
- Active user count

**Technology**: Event streaming (Kafka, etc)

### Batch Metrics

**For standard operational and analytical metrics**:
- Scheduled data loads
- Hourly, daily, or weekly
- Full historical data
- Reliable and predictable
- Simpler architecture

**Examples**:
- Daily sales
- Weekly quality metrics
- Monthly financial results
- Quarterly strategic metrics

**Technology**: ETL jobs (scheduled)

## Data Security

### Access Control

**Restrict access by role**:
- Public dashboards (everyone)
- Department dashboards (department)
- Executive dashboards (leadership)
- Sensitive data (restricted)

**Controls**:
- Authentication (who are you?)
- Authorization (what can you see?)
- Row-level security (data filtering)
- Audit logging (who accessed what?)
- Encryption (data in transit and at rest)

### Data Privacy

**Considerations**:
- Personal data protection (GDPR, etc)
- Sensitive business data
- Competitive information
- Customer information
- Employee information

**Approach**:
- Data classification
- Minimization (only needed data)
- Anonymization (personal data)
- Secure storage
- Regular audits

## Dashboard Integration

### Dashboard Platform

**Technology layer**:
- Visualization tool
- Data connection
- User interface
- Security layer
- Sharing and distribution

**Capabilities needed**:
- Connect to data warehouse
- Support multiple data sources
- Real-time and batch data
- Flexible visualizations
- Performance at scale
- Mobile support
- API access

### Development Process

**Dashboard creation workflow**:
1. Define requirements (metrics, audience, frequency)
2. Design dashboard layout (visual hierarchy)
3. Create data connections (source system links)
4. Build visualizations (charts, gauges, tables)
5. Test and validate (accuracy, performance)
6. Deploy and publish (user access)
7. Monitor and optimize (usage, performance)

### Performance

**Dashboard performance**:
- Page load time (acceptable: <3 seconds)
- Query response time (acceptable: <30 seconds)
- Drill-down speed (immediate)
- Concurrent users supported
- Data freshness vs. performance tradeoff

**Optimization**:
- Pre-calculated metrics
- Caching strategies
- Aggregation levels
- Appropriate visualizations
- Query optimization

## Data Refresh and Updates

### Update Schedules

**Metric refresh by category**:
- Real-time metrics: Continuous
- Operational metrics: Hourly or 4-hourly
- Daily metrics: Daily at standard time
- Weekly metrics: Weekly (e.g., Monday morning)
- Monthly metrics: Early in month (2-5 days)
- Quarterly metrics: Early in quarter

**Publishing**:
- Schedule non-working hours when possible
- Maintain historical versions
- Communicate refresh times to users
- Document any data recalculations

### Data Recalculation

**When to recalculate**:
- Metric definition change
- Data error correction
- Baseline adjustment
- System upgrade
- Process change

**Handling**:
- Maintain historical record
- Document reason for recalculation
- Validate impact
- Notify dashboard users
- Audit trail

## Monitoring Data Health

### Data Monitoring

**Monitor daily**:
- Data load completion
- Row count trends
- Null values
- Outliers
- Data freshness

**Alerts for**:
- Failed data loads
- Data quality issues
- Unexpected changes
- Missing data
- Performance problems

### Troubleshooting

**Investigation process**:
1. Identify the issue
2. Trace to data source
3. Verify calculation
4. Check recent changes
5. Fix root cause
6. Validate correction
7. Document incident
