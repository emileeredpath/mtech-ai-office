## Documentation Standards Philosophy

Quality documentation makes knowledge useful. Poor documentation makes knowledge worthless. Standards ensure consistency, clarity, and usability across all documented knowledge.

**Core Principle**: All documentation should be clear enough that someone unfamiliar with the topic can understand it. If documentation requires explanation, it's not clear enough.

## Documentation Principles

### Clarity

**Documentation is clear when**:
- Understandable to target audience
- Uses plain language
- Avoids jargon (or explains it)
- Well-organized
- Logically flows
- Easy to scan

**Tools for clarity**:
- Short sentences
- Active voice
- Bullet points for lists
- Headers to organize
- Examples to illustrate
- Definitions of terms

### Completeness

**Documentation is complete when**:
- Covers all necessary information
- No gaps that require questions
- Includes context
- Covers edge cases
- Provides examples
- Includes resources

**Test**: Can someone unfamiliar do the task with just this documentation?

### Accuracy

**Documentation is accurate when**:
- Information is correct
- Up-to-date
- Verified by subject matter expert
- Free of errors
- Reflects current reality
- Not outdated

**Maintenance**:
- Regular review
- Quick updates
- Owner verification
- Deprecation of old

### Conciseness

**Documentation is concise when**:
- Includes necessary information
- Excludes unnecessary detail
- Gets to point
- Respects reader's time
- Not verbose
- Not oversimplified

**Balance**: Complete but not overwhelming

### Organization

**Documentation is well-organized when**:
- Logical structure
- Clear hierarchy
- Easy navigation
- Headers guide reader
- Related information together
- Progressive disclosure

### Consistency

**Documentation is consistent when**:
- Same format across similar docs
- Same terminology
- Same structure
- Same style
- Same tone
- Predictable

## Documentation Standards

### Title and Purpose

**Every document includes**:
- Clear title
- Purpose statement (what reader will learn/do)
- Audience (who is this for)
- Date (when created/updated)
- Owner (who maintains it)

**Example**:
"Project Delivery Process for Managers - This document defines how to manage projects from approval through completion. It's for managers overseeing projects. Last updated Jan 2026. Owner: Operations."

### Structure

**Standard document structure**:

1. **Overview/Purpose**: Why this exists
2. **Context**: Background information
3. **Key Concepts**: Important definitions
4. **Process/Details**: Step-by-step or main content
5. **Examples**: Concrete examples
6. **Common Questions**: FAQ
7. **Related Resources**: Links to other docs
8. **Maintenance**: Who updates, when reviewed

### Writing Style

**Documentation style**:
- Second person ("you do X") when instructional
- Active voice ("Do X" not "X should be done")
- Shorter paragraphs
- Shorter sentences
- Simple words
- Avoid jargon
- Numbered steps for procedures
- Bullet points for lists
- Bold for key points

### Formatting

**Standard formatting**:
- Hierarchy: H1 (title), H2 (major sections), H3 (subsections)
- Bold for key terms
- Lists for multiple items
- Code blocks for technical content
- Emphasis (italics) sparingly
- Consistent heading style
- Adequate white space

### Examples and Templates

**Include**:
- Real examples (with actual data)
- Step-by-step examples
- Templates
- Checklists
- Scenarios

**Benefits**:
- Illustrate concepts
- Provide templates
- Make actionable
- Show real usage

### Metadata

**Each document includes**:
- Date created
- Date last updated
- Author/owner
- Related documents (links)
- Tags/categories
- Difficulty level (if applicable)
- Time to complete (if procedural)
- Prerequisites
- Version number (if multiple versions)

## Documentation Types

### Process Documentation

**How to do something**:
- Step-by-step
- Clear sequence
- Decision points marked
- Resources needed
- Time estimate
- Difficulty level

**Example**: Hiring Process, Project Delivery Process

### Policy Documentation

**Rules and expectations**:
- Clear policy statement
- Rationale
- Who it applies to
- Consequences of non-compliance
- Exceptions (if any)
- Who to contact with questions

**Example**: Remote Work Policy, Code of Conduct

### Best Practice Documentation

**What works well**:
- Why this is best practice
- When to use
- Step-by-step approach
- Rationale
- Examples
- Pitfalls to avoid
- Metrics/results

**Example**: Effective Meeting Practices, Customer Onboarding Best Practices

### Lessons Learned Documentation

**What we learned**:
- What happened
- What we learned
- Why it matters
- How we'll apply it
- Changed behavior
- Metrics/impact

**Example**: Project Retrospective, Customer Issue Analysis

### Decision Records

**What we decided**:
- What decision was made
- Why we made it
- Alternatives considered
- Trade-offs
- Who decided
- When decided
- Status/outcomes

**Example**: Tech Stack Selection, Office Location Decision

### FAQ Documentation

**Common questions and answers**:
- Q: [Clear question]
- A: [Clear answer]
- Link to more info if needed
- Index for easy scanning
- Updated frequently

## Review and Maintenance

### Review Frequency

**Documentation is reviewed**:
- Annually (comprehensive)
- When process changes
- When new learning
- When inaccuracy discovered
- Triggered by feedback
- Quarterly (critical docs)

### Deprecation

**Old documentation**:
- Mark as deprecated
- Date of deprecation
- Point to new version
- Archive if historical value
- Remove if no longer needed

### Feedback Loops

**Users can provide feedback**:
- Document is unclear
- Missing information
- Outdated
- Better approach exists
- Questions about document

**Process**: Users comment → Owner reviews → Updates → Shares improvements

## Documentation Quality Assurance

### Review Process

**Before publishing**:
1. Self-review by author
2. Peer review
3. Subject matter expert review
4. Copy editing (if formal)
5. Final approval
6. Publication

### Quality Checklist

**Document meets standards if**:
- [ ] Clear title and purpose
- [ ] Complete information
- [ ] Accurate and current
- [ ] Well-organized
- [ ] Easy to scan
- [ ] Includes examples
- [ ] Links to related docs
- [ ] Owner and date included
- [ ] Appropriate audience level
- [ ] No jargon (or explained)

## Documentation Tools

**Support documentation**:
- Markdown for text
- Google Docs for collaborative editing
- Confluence or wiki for repository
- GitHub for technical docs
- Notion for team knowledge
- PDF for formal documents

**Choose tools based on**:
- Accessibility
- Search capability
- Collaboration needs
- Version control
- Organization needs

## Common Documentation Mistakes

**Avoid**:
- Writing for self, not reader
- Using too much jargon
- Incomplete instructions
- Outdated information
- Poor organization
- No examples
- No owner/date
- Too verbose
- Too technical
- Incorrect format
